BudgeTrak has roughly 45,000 lines across ~93 source files, plus the SSD/LLD, plus 30+ memory files. No single AI session — even with 1M context — can reliably hold the full picture of what's already built. Context compression and session restarts make this worse: by the time the conversation has run for a while, my "memory" of the codebase is actually a lossy summary plus whatever I read recently. That's a recipe for confidently proposing things that already exist, or analyzing problems that have already been solved.

**Why:** This pattern hit four times in a single session on 2026-04-11. Recording all four because they show the failure mode in different shapes.

First, the v2.5.x help-text audit. Earlier rounds had confidently declared the help pages "clean" because they compared the help text to itself or to general keyword searches, rather than tracing each specific claim back to the matching screen file. A more rigorous follow-up pass found six real drifts including stale references to a feature (the manual encrypted save) that had been deleted weeks earlier. The user's exact words: *"we do need to try harder when updating references, memories, and documentation to make sure they all reflect the actual state of the code and our project."*

Second, the OCR / multi-category transactions plan. While brainstorming a subscriber feature for receipt OCR that could split a Target receipt across groceries / household / pharmacy categories, I wrote ~200 lines of architectural analysis comparing three data-model options for "how would BudgeTrak represent multi-category transactions" — recommended a hybrid approach, listed open questions, asked the user to weigh in on the tradeoffs. The user's response was a one-line gut-check: *"Don't we already have multi-category display/entry/editing/charting built in to the transactions list?"* A five-second `grep` for `categoryAmounts` would have shown that **yes** — `Transaction.categoryAmounts: List<CategoryAmount>` has existed in the data model for ages, with full support across 16 files including the dedicated 470-line `PieChartEditor.kt` component, the encryption schema (`enc_categoryAmounts`), the sync merge processor, the dashboard pie chart, the PDF expense report generator, the CSV importer, and the auto-categorizer. Every single thing I was about to propose adding from scratch was already built and shipping.

Third, the privacy policy Crashlytics opt-out claim. While drafting the BudgeTrak privacy policy I wrote: *"Crash reporting can be disabled in your Android system settings under Privacy or Diagnostics."* This is a common misconception about Android's "Usage & diagnostics" toggle, which actually only controls Google's own OS telemetry — it does **not** affect Firebase Crashlytics in third-party apps, because Crashlytics is embedded into the app's own code and only the app can call `FirebaseCrashlytics.setCrashlyticsCollectionEnabled()`. The user caught it with a single question: *"I noticed the privacy policy says users can disable Crashlytics. How do they do that?"* A grep for `setCrashlyticsCollectionEnabled` would have returned zero results in the BudgeTrak codebase, immediately revealing that the claim was false. I'd written a documentation assertion as if it were fact without verifying. We then implemented an actual in-app toggle so the policy could be made truthful instead of just deleted.

Fourth — and this one stings the most because it happened *while implementing the fix for the third* — I was adding a `crashlyticsEnabled` SharedPreferences read in `BudgeTrakApplication.onCreate` and confidently typed `getSharedPreferences("budgetrak_prefs", MODE_PRIVATE)`. The actual prefs name is `"app_prefs"`. A grep for `getSharedPreferences` in `MainViewModel.kt` would have shown the right name in two seconds. I caught it before commit only because I happened to grep one more time before moving on, but the code would have compiled, run, and *silently failed* — every device would have read a fresh empty SharedPrefs file, found `crashlyticsEnabled` missing, fallen back to the default `true`, and effectively ignored every user who tried to opt out. It would have been a phantom feature: visible in the UI, persisted in `app_prefs`, but never actually consulted at startup. The kind of bug that's almost impossible to notice in casual testing.

All four incidents share a root cause: I treated my mental model of the codebase (or of the world generally, in the Crashlytics myth case) as authoritative when it wasn't. Each was "I think this is true, so I'll write it down as fact." Each was wrong. The fourth one is especially worth remembering because it shows the verify-before-proposing reflex needs to apply *during* implementation work too, not just during planning. Even tiny string literals like a SharedPreferences name are code claims that need verification.

**How to apply:**

- **Before proposing any non-trivial code change, design, or architectural recommendation**, run a targeted grep or read for the relevant data type, function name, or feature keyword. Five seconds of verification costs nothing; skipping it costs hundreds of words of wasted plan plus the user's time correcting it. Examples of phrases that should trigger this reflex: "we'd need to add a field for…", "this would require a new data class for…", "we don't currently support…", "BudgeTrak would have to track…".

- **When the conversation summary mentions something existing, verify the citation.** Summaries are written from a snapshot in time and decay as the codebase evolves. Renamed functions, removed flags, changed file paths — all common. If the user is about to act on a recommendation that depends on something the summary mentioned, check the file or grep for the symbol first.

- **Prefer reading actual source files over memory recall** for anything load-bearing. Memory is great for "where do I look" and bad for "what does the code currently do." Use memory as an index, not a database.

- **When the user asks a verifying question** — "don't we already have X?", "did we already do Y?", "is Z still the case?" — treat that as a strong signal that I should drop whatever I was about to say and verify before answering. The user often knows the codebase better than my recent context does, and "I think we don't" is the wrong default response. Right default: "let me check."

- **Before asking the user to make a tradeoff decision**, sanity-check that the tradeoff is real. The hybrid-vs-line-items multi-category data model question wasn't a real tradeoff at all because the answer had been chosen and shipped years ago. The user's time is better spent on actual decisions than on re-litigating settled architecture.

- **Audit prompts to subagents need explicit verification instructions.** A prompt that just says "review this for problems" produces shallow keyword-style results. A prompt that says "verify every specific claim against the matching screen file — cite file:line" produces real findings. See [feedback_help_audit_method.md](feedback_help_audit_method.md) for the help-text-specific version of this lesson.

- **String literals are also code claims that need verification.** When typing a SharedPreferences key, a Firestore field name, a route name, a string resource identifier, or any other "magic string" — grep first, type second. A wrong string literal often compiles fine and silently fails at runtime. The Crashlytics-toggle implementation almost shipped with `"budgetrak_prefs"` instead of `"app_prefs"`, which would have made the entire opt-out feature a phantom that did nothing. Even when you're "just implementing" a known design, the implementation contains its own micro-claims (this prefs name, this function name, this route) that all deserve verification.

- **External-world claims need verification too, not just code claims.** The Crashlytics-via-Android-system-toggle myth wasn't a code question — it was a claim about how Android works. I wrote it into the privacy policy as if I knew it. I didn't. When a documentation claim asserts something about how a third-party system behaves (Android, Firebase, GitHub, Play Store, anything outside this codebase), the same verify-first reflex applies: search the official docs, run the actual experiment, or at minimum hedge the claim. "I'm pretty sure" is not a source citation.

The general principle: BudgeTrak is bigger than my context window even when I think it isn't, and the world outside BudgeTrak is bigger still. Stay diligent. When in doubt, grep — and when grep can't help (because the question is about Android or Firebase or some external system), find a real source instead of guessing confidently.
